# Project Roadmap

> Flutter plugin that turns the rear camera + flash into a contact-PPG heart-rate / RR-interval source for mind_mobile — one pulse source among several, consumed the same way `neiry_kit` is.

> Milestones follow `neiry_kit`'s spine (scaffold → models → API → policy → signal processing → example → service → hardening → integration), decomposed into atomic, independently-revertable tasks. Kit specifics: no vendored SDK; **camera selection is signal-based auto-detect** (note 01); **the Phase-2 spike (GO, note 03) confirmed the kit needs no native code** — enumeration is `availableCameras()` and the torch holds through capture via `setFlashMode(FlashMode.torch)`, so the former native channel-contract and torch-fallback phases are dropped. `flutter_ppg` does the DSP (red-channel, bandpass, peak detection, RR, SQI), so the kit's processing only layers neiry's *acceptance-gate* on top. The one remaining native concern — optional torch-brightness control — is deferred behind the STOP marker.

## Phase 7.5 — De-halving (calibration handoff #1)

> Two on-device runs (67–69 BPM manual reference) exposed **peak-halving**: `flutter_ppg` emits ~2× the real beats, the kit reports 112–131 BPM (≈1.9×), and the neiry-ported acceptance gate *inverts* — its rolling median migrates onto the halved cluster and then rejects the true beats. User constraint: full 15–190 BPM support, **no hardcoded rate cap** (no `minRrMs` floor, no static `PPGConfig.minRRMs`/`minDistance` — a fixed min-distance is itself a max-HR cap). The fix must be **rate-proportional/adaptive**. These tasks sit above the STOP and close the handoff — three orchestrator tasks (design → implement → commit defaults) around one human-in-the-loop calibration.

- [x] **De-halving offline design** — build an offline harness (under `test/` or a throwaway script, NO kit `lib/`/device changes) that replays the two `.calibration/*.json` fixtures through candidate adaptive de-halving algorithms and scores derived BPM vs each file's `manualCount`. Evaluate: (1) RR-domain harmonic-pair merge against an adaptively-tracked fundamental period, (2) driving `flutter_ppg`'s `PPGConfig.minRRMs` from the *tracked BPM* (its built-in adaptivity keys on FPS, not rate — the root cause), (3) waveform-domain fundamental estimation (feasibility only). Pick one; also decide whether `rr_acceptance.dart` still needs median anchoring once de-halving runs upstream. Design-only, no shipped code. Spec: `.ai-factory/notes/29-dehalving-offline-design.md`. [51m 19s]
- [x] **Adaptive de-halving stage** — implement the approach note 29 chose as a pure, unit-tested stage: either a new `lib/src/processing/rr_dehalving.dart` (stateful period-tracker + harmonic-pair merge, placed **before** `RrAcceptance` in `CameraPpgSession._onSignal` so the gate's median only sees de-halved beats, mirroring `RrAcceptance`'s `evaluate`/`reset` shape) OR threading a rate-derived live `minRRMs` into the frame-isolate's `FlutterPPGService`. Rate-proportional only — no constant floor. Regression tests replay both fixtures and assert BPM within counting error; public streams unchanged. Spec: `.ai-factory/notes/30-adaptive-dehalving-impl.md`. [21m 18s]

## Phase 13 — Native camera2 capture path (dual frame source)

> Reverses "no native code" (note 03) **as an addition, not a replacement**: keep the `camera`-plugin capture, and add a native Android camera2 backend that can reach every physical rear lens the plugin hides — notably the near-flash ultrawide, the ideal contact-PPG lens (A70: plugin gives one "0 (unknown)" logical back; zoom is digital-only). Both backends feed the **same** `flutter_ppg` DSP by converging on `FrameMessage` → the existing `FrameIsolate`. The native backend is an **Android-only internal detail** hidden behind a unified camera list; the **user-facing selection + live-preview UX is a public, cross-platform-identical contract** (notes 35/36/42) — because auto-detecting the covered lens is unsolved, the integrator's end-user picks the lens and verifies coverage on the preview, the same way on iOS and Android.

- [ ] **FrameSource abstraction (refactor, no behavior change)** — carve a `FrameSource` seam so `CameraPpgSession` isn't hard-wired to a `camera` `CameraController`. Downstream (isolate, flutter_ppg, de-halving, gate, streams) untouched; keep the close-before-cancel teardown (notes 07/13). Pure enabler, zero behavior change, no native code. Spec: `.ai-factory/notes/39-frame-source-abstraction.md`.
