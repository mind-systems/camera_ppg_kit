# Plan: CameraPpgSession + streams

## Context
Deliver the kit's public measurement surface `lib/src/api/camera_ppg_session.dart`: a wired session that owns a `CameraController` + `FlutterPPGService`, runs the signal-based auto-detect round-trip on `start()`, converts `PPGSignal`→kit models at the edge, and fans out `rrStream`/`qualityStream`/`stateStream` (plus a `[debug]` red-channel stream) as broadcast streams, with the close-before-cancel teardown invariant.

## Settings
- Testing: minimal (one unit test for the pure RR-conversion/dedup helper — it fails silently; camera wiring is hardware-bound and verified on-device via the example)
- Logging: minimal (kit-internal `nlog`-style helper only, per CLAUDE.md — no `print`/`debugPrint`)
- Docs: no

## Assumptions / Scope boundaries
- **No `flutter_ppg`/`camera` type crosses the barrel.** `CameraImage`, `PPGSignal`, `CameraController`, `PPGConfig` stay internal; only kit models (`RrInterval`, `SignalQuality`, `MeasurementState`, `List<double>`) are exposed. This is the load-bearing ARCHITECTURE boundary rule ("no `flutter_ppg`/`camera` type in a public signature", ARCHITECTURE.md line 114) for the milestone.
- **`SignalQuality` name collision (review F1).** `flutter_ppg`'s barrel re-exports its own `enum SignalQuality`; the kit defines its own. `camera_ppg_session.dart` imports both, so `import 'package:flutter_ppg/flutter_ppg.dart' hide SignalQuality;` is mandatory — the kit derives quality via `SignalQuality.fromSnr(signal.snr)` and never reads `PPGSignal.quality`.
- **`start()` returns `Future<CameraPpgError?>` — a deliberate divergence from note 07's `Future<void>` (review F2).** No-finger / camera failure has no `MeasurementState` member (it returns to `idle`, indistinguishable from normal idle), so the typed error must reach the host via the return value — consistent with errors-as-values (ARCHITECTURE.md lines 49/116, return typed values rather than throwing). This changes the frozen signature and diverges from ARCHITECTURE.md line 116's "surface no-finger by emitting a typed state" wording. Reconcile at the Phase-10 freeze: note 07, the note-19 freeze enumeration, **and ARCHITECTURE.md line 116** must be updated to `Future<CameraPpgError?> start()` (error as a returned value), and `mind_mobile` must consume a returned error rather than a state. Flagged here so the freeze audit doesn't read it as a mismatch.
- **The `RrAcceptanceConfig? acceptance` ctor input from note 07 is deferred.** That type is introduced in Phase 6 (note 12); adding it now would pull the acceptance gate forward, which note 07's Guards explicitly forbid ("do not bake acceptance/warm-up logic in yet"). The `debugSignalStream` output *is* in scope (it needs no new type — `List<double>` only).
- **Minimal state machine only.** `idle` when not running, `measuring` while streaming; `warmup`/`done`/`poorSignal` policy is Phase 5 (note 09). This milestone is the bare wired passthrough.
- **Camera override (`availableCameras()`/`useCamera(id)`) is the next milestone (note 08).** This milestone implements only the default auto-detect round-trip inside `start()` so the session functions end-to-end.
- Reference implementations to mirror (kept in the example, not imported): `example/lib/inspector/measurement_runner.dart` (config + bridge + teardown) and `example/lib/auto_detect/coverage_detector.dart` + `camera_probe.dart` (enumeration + coverage round-trip). Port the logic into the kit; do not depend on `example/`.

### Phase 2: Camera acquisition & flutter_ppg bridge

- [x] **Task 3: Rear-camera enumeration + auto-detect coverage round-trip (kit-internal)** (depends on Task 2)
  Files: `lib/src/api/camera_ppg_session.dart` (or a private helper under `lib/src/api/`)
  Port the example's rear-camera enumeration (`availableCameras()` filtered to `CameraLensDirection.back`, probed in enumeration order) and the sequential coverage round-trip into the kit as private helpers. On `start()`: probe each rear sensor in order (torch on, short warm-up + dwell, discard settling frames, coverage = `rawIntensity` within finger-presence bounds over the dwell, threshold ≥ 0.6) and **lock the first covered sensor**. Cameras can't open concurrently — the round-trip is strictly sequential and tears each probe down before the next (never two controllers open). If no sensor is covered, return `CameraPpgError.noFinger` (do not throw — return to `idle`). On a `CameraException` during probing, map via `CameraPpgError.fromCameraErrorCode` and return it. Keep the finger-presence discriminator internal (replicate the bounds check; do not import `flutter_ppg`'s private assessor). The surfaced error reaches the caller via `start()`'s `Future<CameraPpgError?>` return (see Scope note on F2). **Every such early-return (no-finger, probe `CameraException`) must first clear `_running` and reset state to `idle` (review Round-2 F1)** — no-finger is the expected retryable outcome, and leaving `_running` true would make every subsequent `start()` a permanent no-op. The probe tears down its own per-probe controllers internally, so no persistent handle is open on these paths. **Informational (review F8):** the locked camera is torn down at the end of the probe and re-opened in Task 4 — a brief torch off/on that mirrors the spike and preserves "never two controllers open"; not a leak.

- [x] **Task 4: start() — open locked controller, torch, and image-stream bridge into FlutterPPGService** (depends on Task 3)
  Files: `lib/src/api/camera_ppg_session.dart`
  Implement `Future<CameraPpgError?> start()`. **Double-start guard (review F3):** set `_running = true` at the very top of `start()` (before the async probe) and return immediately if it is already true — the four broadcast controllers are created once in the ctor and never nulled, so an example-style `_xController != null` check would be permanently true and must not be copied. This makes a second `start()` during the ~1.1 s/sensor probe a no-op too. **`_running` must be cleared on *every* exit path except a successful lock (review Round-2 F1):** `start()` has three exits — (1) probe locks a covered sensor → stays `_running`/`measuring`; (2) camera-setup below throws → run `_release()`; (3) probe returns `noFinger` or a mapped `CameraException` → cleared per Task 3. Implement this with a single mechanism, e.g. wrap the whole of `start()` (probe + setup) in `try { ... } finally { if (!lockedAndStreaming) _release(); }`, or route all failure returns through `_release()` (which clears `_running` and resets state to `idle`, defined in Task 5). Pick one and name it so no path is left dangling. On the locked camera, create `CameraController(description, ResolutionPreset.low, enableAudio: false, imageFormatGroup: iOS→bgra8888 / else→yuv420)`, `initialize()`, `setFlashMode(FlashMode.torch)`, then best-effort `setExposureMode(locked)` and `setFocusMode(locked)` (each in its own try/catch — focus lock times out on some devices; log and continue). Bridge frames: `_imageStreamCtrl = StreamController<CameraImage>()`; `startImageStream((img){ if(_imageStreamCtrl?.isClosed != true) _imageStreamCtrl?.add(img); })` (use `?.` on the captured field so a late frame after teardown can't crash). Create `FlutterPPGService(config: const PPGConfig())` and subscribe `_sub = service.processImageStream(_imageStreamCtrl!.stream).listen(...)` — the per-signal handler is Task 6. Set state to `measuring` on a successful lock+stream start. Wrap camera setup in try/catch: `CameraException` → `CameraPpgError.fromCameraErrorCode`, other errors → `CameraPpgError.cameraUnavailable`; on failure run `_release()` (Task 5) and stay `idle`.

- [x] **Task 5: stop()/dispose() via a single ordered, idempotent _release()** (depends on Task 4)
  Files: `lib/src/api/camera_ppg_session.dart`
  Implement `Future<void> stop()` and `Future<void> dispose()` delegating to one private `_release()`. Capture-and-null all handles atomically first and clear `_running` so re-entrant calls are no-ops. Order (the spike-proven invariant): 1) `stopImageStream()` if streaming (try/catch); 2) **close the `StreamController<CameraImage>` bridge BEFORE cancelling the `PPGSignal` subscription** — `processImageStream` is an `async*` generator parked on `await for`; cancelling first deadlocks (froze the A70); 3) `await _sub.cancel()`; 4) `service.dispose()`; 5) `setFlashMode(FlashMode.off)` (try/catch); 6) `await controller.dispose()`. `stop()` sets state back to `idle` and leaves the four broadcast controllers OPEN (re-`start()` reuses them). `dispose()` runs `_release()` then closes the four broadcast controllers and marks the session disposed. Ensure no late events are added after controllers close (the `isClosed` guards from Task 6 cover this). **Landed alongside `start()` in Commit 1 (review F6)** so no intermediate commit acquires the camera + torch without a way to release them.

### Phase 3: PPGSignal → kit model conversion at the edge

- [x] **Task 6: Convert each PPGSignal to kit models and fan out** (depends on Task 4)
  Files: `lib/src/api/camera_ppg_session.dart`
  Import flutter_ppg with `hide SignalQuality` (review F1). In the `_sub` handler, convert at the boundary (no `PPGSignal` leaves this method):
  - **RR:** `PPGSignal.rrIntervals` is recomputed from scratch every frame (peaks over a sliding filtered ring buffer → intervals → `OutlierFilter`), not an append-only log — emitting the whole list each time would duplicate. Emit only newly-produced interval(s) via a small pure helper (`_diffNewIntervals(previous, current)`): track the last-seen list and emit the new entries as `RrInterval(intervalMs: rr.round(), timestamp: signal.timestamp, isArtifact: false)` (artifact flagging is Phase 6). **Review F4:** the tail-diff rests on a fragile windowing assumption — values are `seconds*1000*effectiveFPS` and can shift for the same beat, the outlier filter can add/drop entries, and the ring buffer slides front-out/back-in, so raw `double` tail matching may re-emit or misalign. Prefer matching on interval identity/position over raw `double` equality where feasible. Add a prominent comment that this is a minimal passthrough and **real dedup + artifact detection lands in the Phase-6 acceptance gate (note 12)**.
  - **Quality:** `qualityStream.add(SignalQuality.fromSnr(signal.snr))` — the kit's own `SignalQuality`, never `PPGSignal.quality`.
  - **State:** keep minimal — remain `measuring` while streaming (finer `warmup`/`poorSignal`/`done` transitions are Phase 5). Only emit a state change when the value actually changes (don't spam identical events).
  - **Timestamp caveat (review F5):** assigning `signal.timestamp` (frame processing time) to every interval in a batch is a fair approximation for the passthrough but diverges from `RrInterval.timestamp`'s "later peak" contract; `PPGSignal.peakIndices` is available if Phase 6 later wants precise per-peak timing. Note the deviation.
  - **Debug:** tap the same `PPGSignal` and emit its raw/filtered red-channel as `List<double>` on `debugSignalStream` (e.g. `[rawIntensity, filteredIntensity]` or the values note 14's `[debug]` waveform expects — `List<double>` only, never `PPGSignal`).
  - Guard every `add` with `isClosed != true`. Handle `onError` by logging (do not crash the session).

- [x] **Task 7: Unit test the RR diff/conversion helper** (depends on Task 6)
  Files: `test/camera_ppg_session_rr_conversion_test.dart`
  Test the pure `_diffNewIntervals` (extract it as a top-level or `@visibleForTesting` function so it needs no camera). Cover: empty→list, growing window emits only the new entries, unchanged list emits nothing, sliding window (old intervals dropped) does not re-emit already-seen intervals, `double`→`int` rounding. This is the one silent-failure surface in the milestone; camera/stream wiring fails loudly and is verified on-device. **Review F4:** these tests encode the helper's *stated contract*, not proof of correct on-device dedup — keep them scoped to the helper, and treat real-device windowing as a Phase-6 concern.
