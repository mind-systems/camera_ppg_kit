# Plan: Isolate offload for the frame path (defensive)

## Context
Move `flutter_ppg`'s per-frame processing off the UI isolate into one long-lived background isolate per measurement, so a heavy co-tenant screen (the breath-session animation) can't starve the `CameraImage` stream and corrupt the signal. This is a defensive performance change: the public barrel streams (`rrStream`/`qualityStream`/`stateStream`/`fingerPresenceStream`/`debugSignalStream`) and lifecycle stay byte-for-byte identical — it is one internal, revertable concern. Spec: `.ai-factory/notes/13-isolate-frame-offload.md`.

## Settings
- Testing: yes (milestone-mandated: empirical isolate-safety confirmation + unit test of the pure plane→signal path on synthetic frames; no broader test suite)
- Logging: minimal (`nlog` in the kit; `ppgLog`/`ppgTap` in the example per CLAUDE.md)
- Docs: no

## Reconnaissance summary
- **Current frame path** (`lib/src/api/camera_ppg_session.dart`): `start()` bridges `CameraController.startImageStream((img) => imageStreamCtrl.add(img))` → `FlutterPPGService.processImageStream(imageStreamCtrl.stream)` → `_sub.listen(_onSignal)`, all on the UI isolate. `_onSignal(PPGSignal)` reads exactly five fields — `rrIntervals` (`List<double>`), `snr`, `rawIntensity`, `filteredIntensity`, `timestamp` — then converts to kit models via `diffNewIntervals`, `SignalQuality.fromSnr`, `FingerPresence.fromRawIntensity`, `SessionPolicy.onSignal`, `RrAcceptance.evaluate`. Nothing else of `PPGSignal` is used.
- **Teardown invariant** (`_tearDownHandles` / `_release`): stop image stream → **close the `StreamController<CameraImage>` bridge BEFORE cancelling the `PPGSignal` subscription** (`processImageStream` is an `async*` generator parked on `await for`; cancelling first deadlocks) → cancel sub → dispose service → torch off → dispose controller. This ordering must be preserved for the isolate variant and mirrored *inside* the isolate.
- **What must cross the port:** into the isolate, per frame — only the plane bytes `extractRedChannel` reads (`image.planes[0]` for bgra8888; `image.planes[0]` Y + `image.planes[2]` V for yuv420) plus `format.group`/`format.raw` and plane metadata; out of the isolate, per signal — the five `PPGSignal` fields above as plain sendable values.
- **Reconstruction enabler (confirmed by source read):** `package:camera`'s `CameraImage.fromPlatformData(Map)` is a public but `@Deprecated` constructor that rebuilds a `CameraImage` from a plain map (`bytes`, `bytesPerRow`, `bytesPerPixel`, `width`, `height`, `format` raw code). It raises `deprecated_member_use` under `flutter_lints`, so the reconstruction line needs an `// ignore: deprecated_member_use`. Its `format.group` resolution (`_asImageFormatGroup`) reads `defaultTargetPlatform` (which resolves correctly inside a spawned isolate via `Platform.operatingSystem`) and only maps yuv420/bgra8888/jpeg/nv21, so the exact raw `format` code must be passed — **Android yuv420 = `35`, iOS bgra8888 = `1111970369`**. `flutter_ppg`'s `SignalProcessor.extractRedChannel` reads only `format.group` + plane bytes, so a correctly-reconstructed image drives it. This makes **variant (a)** — run the whole `FlutterPPGService` inside the isolate — the viable shape.
- **yuv420 plane indices must be preserved (Android silent-failure trap):** `SignalProcessor._extractRedFromYUV420` indexes `image.planes[0]` (Y) and **`image.planes[2]`** (V) directly. A reconstructed 2-element `[Y, V]` planes list makes `planes[2]` a `RangeError`, which `processImageStream` swallows (`try { extractRedChannel } catch { continue; }` at `flutter_ppg_service.dart:165`) — so **every Android frame is skipped, no `PPGSignal` is emitted, and no error surfaces**. iOS (single bgra8888 plane 0) is unaffected, making this trivial to miss on an iPhone. The isolate must rebuild a **3-element** planes list with V at index 2 and a placeholder at index 1 (`extractRedChannel` never reads index 1); the placeholder still needs a non-null `bytesPerRow` and a (possibly empty) `bytes`, since `Plane._fromPlatformData` requires `bytesPerRow as int`.
- **Transfer copy nuance:** `TransferableTypedData.fromList([...])` copies the bytes into an internal buffer at *construction*; the win is on *transfer/materialize* across the port (no re-serialization, no second copy on receive) — cheaper than a map/JSON round-trip, but not literally "zero-copy per frame." Don't rely on it for a hard per-frame byte-budget claim.
- **Why not variant (b):** `flutter_ppg` exposes only `processImageStream(Stream<CameraImage>)` publicly; its internal reduce→filter→peak stages are private, so a pre-reduced `double` cannot be fed into the public API. Variant (a) is the design; variant (b) is only a fallback if Task 1 disproves (a).
- **Auto-detect probe path** (`_probeCameraCoverage` / `_lockCoveredCamera`): short-lived per-sensor round-trip, not the FPS-sensitive sustained-measurement path. It stays on the UI isolate — out of scope for this offload.

## Architecture note (dependency-rule reconciliation)
ARCHITECTURE.md rule 4 says `src/processing/` imports `src/models/` only (no `camera`/`flutter_ppg`), and note 13 places the isolate entrypoint in `processing/`. For variant (a) the entrypoint necessarily imports `flutter_ppg` (→ `camera`), so those two cannot both hold literally. Reconciliation used by this plan:
- **Pure, sendable value types** (`FrameMessage`, `SignalMessage`) live in `src/processing/frame_message.dart` — `dart:typed_data` + `dart:isolate` (for `TransferableTypedData`, which is declared in `dart:isolate`, **not** `dart:typed_data`), no `camera`/`flutter_ppg`/Flutter imports. `dart:isolate` is pure Dart, so the file stays hardware-free, isolate-runnable, and unit-testable. Rule 4 preserved here.
- **The camera/flutter_ppg-aware isolate host + entrypoint + `CameraImage`↔`FrameMessage` adapters** live in `src/processing/frame_isolate.dart`, documented as the *single* `processing/` file permitted `camera`+`flutter_ppg` imports because it is the isolate-boundary host. The imports it pulls in are pure-Dart data paths (`CameraImage.fromPlatformData`, `FlutterPPGService`), not plugin channels, so the file remains runnable in a spawned isolate and in tests — the spirit of rule 4 (hardware-free, isolate-safe) is intact. The `CameraController` wiring stays in `src/api/` per note 13. Record this deliberate exception in ARCHITECTURE.md (as part of Task 4) so the next reader is not surprised by a `flutter_ppg` import under `processing/`. (An equally clean alternative — hosting `frame_isolate.dart` under `src/api/`, which is already permitted those imports, keeping `processing/` literally pure — was considered; this plan chooses the `processing/` placement per note 13.)

### Phase 2: Isolate frame-offload machinery

- [ ] **Task 2: Sendable message value types** (depends on Task 1)
  Files: `lib/src/processing/frame_message.dart`
  Pure Dart — imports `dart:typed_data` + `dart:isolate` only (`TransferableTypedData` is declared in `dart:isolate`), no `camera`/`flutter_ppg`/Flutter imports. Define `FrameMessage`: the transferable planes (`List<TransferableTypedData>` — only the planes `extractRedChannel` needs) plus `width`, `height`, per-plane `bytesPerRow`/`bytesPerPixel` (ints), and the raw platform `format` code (int) so the isolate can rebuild `format.group`. Preserve enough plane structure to rebuild the **original plane indices** (see Task 3): for yuv420 the message must let the isolate place V at index 2. Define `SignalMessage`: `List<double> rrIntervals`, `double snr`, `double rawIntensity`, `double filteredIntensity`, `int timestampMicros` — the exact sendable subset `_onSignal` consumes, nothing more (no `PPGSignal`, no `CameraImage`). Add a `SignalMessage.error`-style variant or a nullable error field so isolate-side failures cross as **data, not thrown exceptions** (mirrors the no-exceptions-across-the-boundary rule).

- [ ] **Task 3: `CameraImage` ↔ `FrameMessage` adapters** (depends on Task 2)
  Files: `lib/src/processing/frame_isolate.dart` (new)
  Two functions in the isolate-boundary file (may import `camera`): `frameMessageFromCameraImage(CameraImage)` — producer side, extracts only the required plane(s) by `format.group` (bgra8888 → plane 0; yuv420 → Y plane 0 + V plane 2), wraps each plane's `Uint8List` as `TransferableTypedData.fromList([bytes])` (copies at construction; the win is the cheap cross-port transfer/materialize, not a per-frame zero-copy), and carries `format.raw` + per-plane `bytesPerRow`/`bytesPerPixel` metadata; and `cameraImageFromFrameMessage(FrameMessage)` — isolate side, materializes the transferables and rebuilds a `CameraImage` via `CameraImage.fromPlatformData({...})` (with `// ignore: deprecated_member_use`). **Plane-index preservation (Android silent-failure trap):** for yuv420 the rebuilt `planes` list must be **3 elements** with the V bytes at index 2 and a placeholder at index 1 (non-null `bytesPerRow`, empty `bytes`) — a 2-element `[Y, V]` list makes `extractRedChannel`'s `planes[2]` throw a `RangeError` that `processImageStream` silently swallows, producing no signal on Android. Pass the correct raw `format` code (Android yuv420 = `35`, iOS bgra8888 = `1111970369`) so `_asImageFormatGroup` resolves the group. Keep both functions allocation-lean (per-frame hot path).

- [ ] **Task 4: Long-lived isolate host + entrypoint** (depends on Task 3)
  Files: `lib/src/processing/frame_isolate.dart`, `.ai-factory/ARCHITECTURE.md`
  Also add a one-line note to ARCHITECTURE.md's Dependency Rules recording that `src/processing/frame_isolate.dart` is the deliberate, sole exception to rule 4 — it imports `camera`+`flutter_ppg` because it is the isolate-boundary host.
  Add a `FrameIsolate` class and a top-level entrypoint function. On `start()`: `Isolate.spawn` **one** long-lived isolate (never `Isolate.run` per frame — that would re-spawn 24–60×/s), complete a `SendPort` handshake, and expose `Stream<SignalMessage>` (broadcast) plus `void sink(FrameMessage)` and `Future<void> dispose()`. Inside the entrypoint: feed incoming `FrameMessage`s through `cameraImageFromFrameMessage` into a `StreamController<CameraImage>` piped to `FlutterPPGService.processImageStream`, map each `PPGSignal` → `SignalMessage`, and `send` it back over the reply port. On shutdown, mirror the close-before-cancel invariant **inside** the isolate (close the `CameraImage` controller before cancelling the `processImageStream` subscription) before the isolate exits, then `Isolate.kill`. Surface entrypoint errors as `SignalMessage` error values, never thrown across the port. Use `nlog` for coarse milestones only (spawn/handshake/teardown), not per frame.

### Phase 4: Verify

- [ ] **Task 7: Unit-test the pure plane→signal path on synthetic frames** (depends on Task 3)
  Files: `test/frame_transfer_test.dart` (new)
  No hardware. Construct synthetic yuv420 (Y + V planes) and bgra8888 (single plane) byte buffers, round-trip them through `frameMessageFromCameraImage` → `TransferableTypedData` → `cameraImageFromFrameMessage`, and assert the rebuilt `CameraImage`'s planes/format reproduce the input. **Assert the yuv420 rebuild has `planes.length == 3` and `planes[2]` carries the V bytes** (guards the Android plane-index trap from Task 3). Verify the red-channel mean matches an inline reference computation — **not** `extractRedChannel` (it lives in `signal_processor.dart`, which the `flutter_ppg` barrel does not export, so it is not callable from the test); replicate the formula: bgra8888 = mean of every 4th byte from offset 2; yuv420 = `Mean(Y) + 1.402 * (Mean(V) - 128)` over **all** plane bytes (including any row padding). Cover the empty-plane and single-pixel edge cases. This locks the boundary contract without a device.
